# 🧠 Informe Técnico Completo y Justificado - Avatar Interactivo

## 📌 Objetivo del Documento

Este documento describe en profundidad TODAS las mejoras, decisiones técnicas y cambios realizados en el sistema del avatar interactivo.

Su propósito es que cualquier integrante del equipo o evaluador pueda:
- **Entender la lógica completa del sistema**, desde la captura de voz del usuario hasta la animación del avatar.
- **Comprender el "por qué"** de cada decisión técnica tomada durante el desarrollo.
- **Saber cómo mantener, modificar o escalar el proyecto** en el futuro.
- **Tener material sólido para defensa académica**, justificando las mejoras en rendimiento y UX.

---

## 🏗️ Arquitectura General del Sistema

El sistema opera bajo un modelo cliente-servidor, donde el Frontend (React) gestiona la interfaz, la interactividad y la captura de audio, mientras que el Backend (Node.js + Express) funciona como un orquestador de intenciones, datos y TTS.

### 🔄 Flujo Completo de la Aplicación

A continuación, se detalla el ciclo de vida completo de una interacción:

```text
[ Usuario ]
   │
   ▼ Habla por micrófono
[ Speech To Text (Navegador nativo en App.js) ]
   │
   ▼ Transcribe texto tras un delay para evitar cortes
[ Frontend (React) ]
   │
   ▼ Envía el texto plano al endpoint /chat
[ Backend (Node.js + Express - index.js) ]
   │
   ▼ Delegación a Sistema de Capas en Cascada
   ├── 1. Caché: ¿Se preguntó lo mismo recientemente? -> Si sí, retorna respuesta.
   ├── 2. FAQ (helpers/faq.js): Busca coincidencias exactas y predefinidas.
   ├── 3. Parser (helpers/parser.js): Extrae entidades concretas (ej. materias, horarios).
   ├── 4. Clasificador (helpers/intencion.js): Enruta a la IA o funciones específicas.
   └── 5. IA Fallback (helpers/Ollama.js): Genera respuesta dinámica si no hay match anterior.
   │
   ▼ Se genera una respuesta de texto
[ Text To Speech (Piper - helpers/tts.js) ]
   │
   ▼ Genera archivo de Audio (.wav) y sincronización de labios (Mouth cues / Blendshapes)
[ Backend retorna Audio + Texto + Blendshapes ]
   │
   ▼ 
[ Frontend (React - Avatar.jsx y App.js) ]
   │
   ▼ Reproduce Audio + Actualiza Estado (ej. 'talking')
[ Avatar 3D (Three.js / React Three Fiber) ]
   │
   ▼ Ejecuta animación FBX, Lip-Sync procedural y micro-movimientos.
[ Usuario (Escucha y Ve al Avatar) ]
```

## ⚙️ BACKEND - OPTIMIZACIÓN Y LÓGICA

### 🎯 Problema Inicial
El sistema original dependía exclusivamente de la Inteligencia Artificial (LLMs) para procesar *cada una* de las intenciones del usuario. Esto generaba:
- Alta latencia (6–10 segundos por respuesta).
- Consumo innecesario de recursos.
- Respuestas poco controladas, con riesgo de alucinaciones en preguntas sencillas o reglamentarias.

---

### 🧠 Solución: Arquitectura en Cascada (5 Capas)
Se abandonó el enfoque "IA para todo" a favor de un sistema de resolución determinista con IA como último recurso.

Se implementó un sistema en orden estricto de prioridad (ver `index.js` y llamadas correlativas):
1. **Caché en memoria**: Respuestas instantáneas para preguntas idénticas repetidas en la sesión.
2. **FAQ (respuestas predefinidas)** (`helpers/faq.js`): Respuestas cerradas para preguntas de rutina de la facultad.
3. **Parser (datos estructurados)** (`helpers/parser.js`): Extrae entidades de los JSON de datos (ej.: "A qué hora es X materia").
4. **Clasificación de intención** (`helpers/intencion.js`): Intermediario que decide qué herramienta invocar.
5. **IA (fallback)** (`helpers/Ollama.js`): Ollama local entra solo si las capas 1 a 4 no devolvieron un resultado satisfactorio.

**✅ Justificación:** 
- Reduce drásticamente el uso de la IA local.
- Mejora la velocidad de respuesta para preguntas frecuentes.
- Aumenta el control sobre respuestas (crucial en entornos institucionales para no dar información errónea).

---

### ⚡ Caché en Memoria
- **Implementación**: Uso de un `Map()` en memoria volátil de Node para almacenar el par ordenado `{ "pregunta": "respuesta_procesada" }`.
- **Justificación**: Evitar reprocesar consultas idénticas (muy común en stands de exposición).
- **Resultado**: Las respuestas "cacheadas" se resuelven en ~0ms, ahorrando ciclos de CPU valiosos que hubiesen recaído en el modelo LLM y el motor de Piper TTS si el audio también se cachea.

### 🧠 Clasificador de Intención (`helpers/intencion.js`)
- **Implementación**: Es un subsistema determinista que parsea el string de entrada buscando palabras clave estratégicas usando Regex/Arrays antes de invocar la IA.
- **Justificación**: Resulta un desperdicio encender un modelo complejo para interpretar la pregunta "¿Qué materias hay?". 
- **Impacto**: Actúa como un router rápido. Permite enviar la solicitud hacia `parser.js` para búsquedas en la base de datos, `faq.js` para respuestas estándar, o `Ollama.js` para inferencia libre, optimizando los tiempos.

### 📦 Modularización de Datos
**Antes**: Se utilizaba un único archivo JSON ("la base de datos") gigantesco que agrupaba toda la información.
**Ahora**: Se han desglosado en archivos especializados dentro de la arquitectura o en un esquema interno manejable (ej. `helpers/facultad_db.json` con secciones claras y documentadas como `materias`, `ubicaciones`, etc).
- **Justificación**: 
  - Evitar cargar datos innecesarios en memoria indiscriminadamente.
  - Facilitar el mantenimiento por parte del equipo.
  - Permitir escalar el proyecto (por ejemplo, añadir entidades para docentes o bibliografía futura) de forma predecible sin dañar el parseo general.

### 🤖 IA Controlada (Ollama local - `helpers/Ollama.js`)
- **Problema Inicial**: El uso excesivo generaba una latencia altísima incompatible con la interactividad en tiempo real (6-10s cortan la charla).
- **Solución**: 
  - Restricción estricta de parámetros en la request al modelo: `num_ctx` limita la cantidad de tokens guardados en memoria del diálogo (contexto). `num_predict` acorta forzosamente la respuesta saliente (evitando largos y aparatosos discursos generados).
  - Configurado estrictamente como método auxiliar de "Fallback", invocado cuando el input no hace match con reglas estáticas (`faq.js`, `parser.js`).
- **Justificación**: Optimiza la performance en hardware mediano-bajo logrando el equilibrio fundamental del IA asistencial: razonamiento versus agilidad.

### 📊 Logging de Performance
Se han integrado mediciones de tiempo (`console.time / console.timeEnd`) a lo largo de las fases de backend:
- Tiempo total (RTT) devuelto al cliente y render.
- Capa resolutiva interviniente ("Servido desde FAQ", "Capa IA llamada").
- Demora pura de IA (útil para perfilar modelos pesados vs rápidos).
- **Justificación**: Poseer métricas sólidas y defendibles para la presentación ("El sistema mejoró su respuesta global de 6s a 800ms") y un panel vital para que futuros desarrolladores cacen cuellos de botella.

## 🎨 FRONTEND - UX Y COMPORTAMIENTO (React / Three.js)

### 🎯 Problema Inicial
Antes de la intervención, la capa cliente evidenciaba problemas como:
- **UX robótica y pasiva**: Avatar duro y sin feedbacks para el usuario.
- **Cortes de la voz prematuros**: El sistema interrumpía al humano cuando éste pausaba al hablar.
- **Animaciones "rotas"**: Movimientos del modelo fuera del marco central (Root Motion problem).
- **Sensación de "Cuelgue"**: Periodos ciegos mientras Node.js trabajaba los datos en su servidor.

---

### 🎤 Mejora Captura de Voz (STT) (`src/App.js`)
- **Problema**: El sistema nativo de dictado asumía el silencio como cierre y cortaba frases frecuentemente (Ej: "A qué hora es... Física" resultaba en un request de solo "A qué hora es").
- **Solución**: 
  - Se agregó un sistema tolerante de debounce/delay de ~800ms antes de procesar el string y dispararlo al endpoint.
  - Lectura continua usando `interimResults: true` si es aplicable para mostrar que se está escuchando asincrónicamente.
- **Justificación**: Respeta el ritmo orgánico del pensamiento humano; da el espacio biológicamente normal a los baches respiratorios o cognitivos al hablar.
- **Resultado**: Incremento masivo de queries acertadas al haber recepcionado correctamente oraciones completas.

### ⏳ Mejora Simulación de Pensamiento
- **Problema**: Al mandar la request, la máquina quedaba "congelada" durante los ~1 a 2 segundos de red local o remota.
- **Solución**: 
  - Engatillado del estado `thinking` de forma imperativa en `App.js` apenas viaja el prompt.
  - Intercalado de Delay artificial o forzado para rutinas asíncronas, garantizando exhibir la animación de análisis por una fracción de tiempo.
- **Justificación**: Principio clave de las interfaces: "La espera informada y animada se siente más corta que una pantalla congelada interactuando". Refuerza la analogía con una persona deliberando sobre lo que se le preguntó.

### 🧠 Máquina de Estados Finita (`src/App.js`)
Se reordenaron todos los estados reactivos, agrupando la representación conductual del bot bajo valores de variables rigurosamente delimitados en el hook principal del ciclo vital del avatar:

```text
       [ Idle (Inactivo, quieto) ]
               │
    (Usuario comienza a hablar)
               ▼
[ Saludo ] ───▶ [ Listening (Prestando atención) ]
                       │
             (Micrófono entra en delay final)
                       ▼
               [ Thinking (Mano al rostro / pensativo) ]
                       │
             (Respuesta servida desde Backend)
                       ▼
               [ Talking (Moviendo brazos, gesticulando) ]
                       │
     (Terminó buffer de audio y LipSync procedural)
                       ▼
               [ Regresa a Idle ]
```
- **Justificación**: Ordena el código de UI, haciéndolo autoexplicativo (self-documenting code) protegiéndolo de que dos componentes colisiones (ej: no se enviarán requests de voz mientras `avatarState === 'talking'`).
- **Impacto**: Lógica visual fluida. El código se torna infinitamente más transparente para programadores nuevos adentrándose a trabajar con el proyecto.

---

### 🎭 Sistema de Animaciones (`src/avatar/Avatar.jsx` y `config.js`)
- **Problema**: Faltaban asociaciones lógicas que empaten al avatar en un entorno comunicativo (explicaba información mirando al vacío como estatua).
- **Solución**: El dictamen un "Mapping de Inferencia" en Frontend basado en el contexto de la intención entregada en el endpoint o estado en curso para enlazar key strings (`"thinking"`, `"idle"`) a archivos `.fbx` puntuales.
- **Justificación**: Separa el motor de lógicas puras de la representación de Three.js (Implementación modular en Capa de Vista pura).

### 🎬 Eliminación de Root Motion ("Problema de Anclaje")
- **Problema**: Animaciones previas de repositorios (mixamo/etc) contenían variables de movimiento global en el Track0 ("Root node motion"); esto provocaba que cada vez que el avatar concluía una animación, quedaba ligeramente descentrado, hasta eventualmente salirse y abandonar el ángulo útil de la cámara 3D focalmente establecida.
- **Solución**: Borrado y barrido exhaustivo de las pistas de la propiedad `.position` en los nodos maestos de FBX (sea manipulado el asset, o mitigado programáticamente descartando keys con Threejs).
- **Justificación**: Robustez de formato. Evita la sobre-intervención de las cámaras de código, manteniendo un encuadre perfecto.

### 🎞️ Transiciones Suaves
- **Implementación**: Se aplica profusamente interpolación cruzada (`crossfade`) nativa del `AnimationMixer` al dictar cambio entre animaciones contiguas.
- **Justificación**: Saltar inminentemente de un frame de Idle a un talk sin blending produce una brusca rotura muscular biomecánica en la vista. Realizar el traspaso interpola rotaciones en un deltaT evitando el resalto sintético, garantizando un salto orgánico e inapreciable entre poses extremas.

### 👄 Lip Sync Procedural y Biomecánica Minorista (`helpers/blendshapeNames.js` y componente React)
- **Problema**: Modelos Piper sin pre-render fallaban en hacer Lipsync 100% per-phonema certero o el avatar presentaba un efecto estatua perjudicial ("Uncanny Valley").
- **Solución**: 
  - Uso de Blendshapes (Morph Targets definidos) con intensidades y mapeo probabilístico pseudo-aleatorio atado a la longitud del buffer y amplitud sonora, operando un render procedural iterativo.
  - Implementación de Micro-movimiento en zona ocular (parpadeo en loop `setInterval` / `setTimeout` no correlacionado y dinámico insertado de forma nativa a Threejs).
  - `Idle` dinámico (alteraciones ligerísimas de la tasa de animación original).
- **Justificación**: Un ser vivo pestañea y procesa microsaccades biológicos. Reproducir esos rastros vitales de manera económica (matemáticamente) asegura que los humanos miren al Bot sin percibir aversión visual ni sentirse ansiosos.

---

### 🔁 Sistema de Sesión Auto-reset (`src/App.js` con hooks de Inactividad)
- **Implementación**: Un listener pasivo de audio que ejecuta un `setTimeout` superior a 10–15 segundos. Si en ese periodo no hubiese señal al STT, se reinicia todo estado, caché base y banderas del front.
- **Justificación**: Arquitectura construida en base al patrón "Kiosk-Mode" o Totem Stand-Alone. Los usuarios que lleguen e interactúen con el totem no deben encontrar el estado "en medio de charlar" derivado del humano anterior que abandonó la computadora o display físico. Este "limpiaparabrisas" lo asienta universalmente para servir de 0.

## 🛠️ GUÍA DE MANTENIMIENTO TÉCNICO

*   **¿Añadir nueva información curricular / institucional?**
    ➡ Dirígete a cargar data y a revisar el array manejador en base al json primario o `helpers/facultad_db.json`. Testear en `helpers/parser.js` que el mapping esté contemplado de estar enrutado por la capa estricta.
*   **¿Agregar una nueva animación o gesto?**
    ➡ Depositando el modelo procesado en `public/animations/`, asegurando matchear su nombre clave en el listado mapeado sobre `src/avatar/config.js` y usándolo convencionalmente mediante string keys para solicitar su disparo.
*   **¿Ajustar el FAQ de chistes rápidos o consultas reiterativas?**
    ➡ `helpers/faq.js`. Esta capa intercepta todo query repetido evitando procesamiento LLM costoso.
*   **¿Modificar la sensibilidad de habla, o retardos?**
    ➡ Hook principal de ciclos en componentes `src/App.js` englobando timeouts e iteradores de STT.

## ⚠️ DECISIONES ARQUITECTÓNICAS CLAVE (Resumen para Defensa Académica)

1. **MINIMIZAR EL ABUSO DE IA LIMITANDO AL FALLBACK**: Decisión angular validando que LLM no es base de datos.
2. **PRIORIDAD ABSOLUTA DEL RENDIMIENTO**: En tiempo y CPU limitados, la información determinista de `faq.js` (retornando a 0ms) supera mil discursos lentos a 10s por parte de Modelos de Lenguaje.
3. **UX SOBRE FUERZA BRUTA DE HARDWARE**: Utilizar estados cognitivos explícitos ("Thinking/Listening") anula virtualmente el impacto negativo del delay real en los request; aplicando ciencia cognitiva de interfaz humana para retener empatía interactor-avatar.
4. **SISTEMA MODULAR / MVC-FRIENDLY**: Separación irrefutable de la manipulación semántica (parser) de la manipulación volumétrica visual (React Three Fiber / config animations), resultando en código inmaculadamente mantenible a futuro.
